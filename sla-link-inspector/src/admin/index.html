SLA Alert Settings

How It Works

Alerts use the parent ticket's SLA. Comments and @mentions go on linked tickets.

  • Triggers — The parent ticket’s SLA (at risk, breached, or low time left).
  • Recipients — Who to @mention on each linked ticket.
  • Delivery — Jira, Slack, and/or email — use any mix.

1 SLA Field

SLA lives on the parent ticket. Leave blank to auto-detect the field.

Jira: Project settings → Fields. JSM SLA: usually leave blank.

2 Alert Triggers & Recipients

Configure when alerts fire and who gets notified on linked tickets.

Fires when the parent SLA has 25% or less of its total time left (not yet breached). Jira may label this “at risk” in the UI.

Fires when the parent SLA time has expired.

Skip alerts when the linked issue is done.

3 Warning Thresholds

A one-time alert fires when the SLA crosses each threshold — before or after breach. Each threshold fires independently. Choose who to notify per row.

Use a negative number to alert when breached (e.g. -1 days = 1 day past due).

4 Where Alerts Go

Turn on what you need. Jira comments always go on linked tickets.

Jira

Includes the parent ticket’s SLA summary.

Off: comment only, no @mention.

Slack

Choose one or more Slack delivery methods.

Channel Via Webhook

Incoming Webhook URL from Slack.

If both webhook and channel ID are set, channel posts use the webhook.

Channel Via Bot

Invite the bot to the channel first.

Direct Messages Via Bot

DMs use Jira email when available, or a configured Slack member ID per Jira user.

If Post a Jira comment and @mention recipients are on (defaults), assignee/reporter/etc. get a Slack DM automatically when a bot token is saved—you do not need this box unless you turned mentions off and still want DMs.

These addresses receive a DM for every SLA alert, regardless of which ticket triggers it. Use for team leads or on-call contacts who monitor all SLAs.

When Jira does not expose email, DMs still work if the user's Slack member ID is listed here. Admin rows override self-service entries for the same Jira account.

Slack App Permissions (Admins)

DMs need im:write (and chat:write). Email lookup also needs users:read.email. Member-ID DMs work without users:read.email for mapped users.

Email

Use Zapier or Make to forward into your inbox.

Payload includes issue key, SLA status, and message text.

Slack bot — Not needed for webhook-only delivery. Add channel via bot (without webhook) or turn on DMs to configure the token and tests.

5 Slack Bot Setup

Use when you post to a channel via the bot or send DMs. Not needed for webhook-only delivery.

Updating the token? Clear saved, then paste the new one.

Where to Find the Token

Slack app → OAuth & Permissions → Bot User OAuth Token.

Test Slack Delivery

Channel post uses webhook if set; otherwise bot and channel ID. Test DM: first email under “Always notify these emails” if set; otherwise optional issue key (assignee); otherwise the Jira user running the test.

Automated Alerts

Same text for Jira comments and Slack/email on automated triggers.

Placeholders

  • {{issueKey}} — parent (SLA ticket)
  • {{slaName}} — SLA name
  • {{remainingTime}} — e.g. 9d left
  • {{status}} — at risk / breached / in range
  • {{priority}} — ticket priority (e.g. High)
  • {{assignee}} — @mentions configured in Alert Triggers & Recipients

~500 characters or less reads well in Jira.

Manual Send

When someone uses “Send SLA Alert” on an issue.

Placeholders

  • {{issueKey}} — parent SLA ticket
  • {{slaStatus}} — At-Risk / Breached / in range
  • {{priority}} — parent SLA ticket priority
  • {{expiryDate}} — expiry
  • {{assignee}} — @mentions configured in Alert Triggers & Recipients

~500 characters or less reads well in Jira.
